= ADR 0018 Worker-Push Segment Data-Plane Check

== Question

Does the implemented worker-push path remain practical when segment bytes use a dedicated gRPC
HTTP/2 stream on the worker's existing mTLS channel?

== Method

On 2026-07-13, a disposable integration test exercised the production client and server upload
code over loopback with 64 KiB data frames. Timing started before dispatch and ended when the
atomically stored segment became visible through `LocalSegmentStore`, so it included command
delivery, fake FFmpeg publication, gRPC upload, validation, copying, and the filesystem write.

[cols="1,1,1"]
|===
|Segment |Elapsed |Effective throughput

|512 KiB
|0.192 s
|2.6 MiB/s

|4 MiB
|0.109 s
|36.6 MiB/s

|16 MiB
|0.111 s
|144.7 MiB/s
|===

== Conclusion

Proceed with the bounded worker-push path. It does not reproduce the earlier server-pull
microbenchmark's 345+ MiB/s result, and the ascending run gives later samples a JIT-warmup
advantage, so these numbers are not an absolute transport ceiling. They do show that fixed
per-upload latency dominates small segments while bulk throughput remains comfortably above
realistic media bitrates. The normal six-second segment cadence leaves substantial headroom.

The production path therefore keeps 64 KiB frames, a 16 MiB segment limit, one dedicated stream
per upload, exact declared-length validation, and atomic publication. Re-benchmark on representative
networked hosts before raising the limit or changing the transfer model.
