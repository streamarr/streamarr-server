= Architecture Overview

== Tech Stack

* **Java** (LTS), **Spring Boot**, **PostgreSQL**
* **GraphQL** via Netflix DGS
* **jOOQ** for complex queries (keyset pagination)
* **Flyway** for database migrations
* **FFmpeg** via ProcessBuilder for HLS transcoding

== Layered Architecture

----
Protocol Adapters (GraphQL Resolvers, REST Controllers)
        |
    Services (business logic)
        |
    Repositories (data access)
        |
    Domain Entities (core model)
        |
    PostgreSQL (via Flyway migrations)
----

*Dependency rule:* each layer only depends on the layer below it.

* *Protocol adapters* (GraphQL resolvers, REST controllers) delegate to services -- no business logic at the protocol layer
* *Services* orchestrate business operations -- protocol-agnostic (no `graphql.*` imports, enforced by ArchUnit)
* *Repositories* use JPA for CRUD and jOOQ for complex queries
* *Domain* depends on nothing -- pure data model

These rules are enforced by ArchUnit tests (see link:adr/0014-protocol-agnostic-pagination.adoc[ADR 0014] for pagination-specific architecture).

== Database

PostgreSQL with UUID primary keys, custom ENUMs, and audit columns (`created_on`, `modified_on`, `created_by`, `modified_by`) on all entities.

Entity hierarchy uses JPA JOINED inheritance:

----
BaseAuditableEntity (UUID PK, audit columns)
    └── BaseCollectable (title, summary, release_date)
            ├── Movie
            └── Series
----

== GraphQL API

Netflix DGS with Relay-style cursor-based pagination:

* `MediaConnection` / `MediaEdge` / `PageInfo`
* Keyset (seek) pagination via jOOQ for performant infinite scroll

== Metadata Pipeline

----
Library Scan
    → Walk filesystem
    → Parse filename (regex-based, 59+ test cases)
    → Search TMDB for match
    → Enrich with TMDB metadata (cast, studios, ratings)
    → Persist Movie + MediaFile + relationships
----

The metadata provider abstraction (`MetadataProvider<T>`) allows swapping TMDB for another source without changing the scan pipeline.

== Service Communication

Services coordinate library lifecycle events using Spring Application Events rather than direct method calls. `LibraryManagementService` publishes events; interested services listen and react independently.

----
addLibrary()
    → LibraryAddedEvent
        → DirectoryWatchingService (register watcher)

scanLibrary()
    → ScanCompletedEvent
        → OrphanedMediaFileCleanupService (remove orphaned files)

removeLibrary()
    → LibraryRemovedEvent
        → DirectoryWatchingService (unregister watcher)
        → StreamingSessionCleanupListener (terminate sessions)
----

Side effects triggered by transactional operations use `@TransactionalEventListener(AFTER_COMMIT)` to ensure they only execute after a successful commit. Non-transactional publishers use `@EventListener` for synchronous, same-thread delivery.

See link:adr/0010-spring-application-events.adoc[ADR 0010] for full rationale.

=== Distributed Transcoding Direction

link:adr/0018-live-playback-authority-and-outbound-workers.adoc[ADR 0018] keeps local playback as the default and preserves independent rendition failure through one shared FFmpeg engine.
An optional worker opens an outbound, long-lived mTLS gRPC `WorkerSession`; the live connection owns registration, capacity heartbeats, commands, and job-attempt fencing.
Segment uploads use separate bounded HTTP/2 streams over the same channel, and media is addressed by opaque `MediaSourceRef(source_namespace_id, relative_key)` values rather than host paths.
The initial topology is one Streamarr server and zero or more workers; workers have no database access or inbound listener, and multi-server coordination is deferred.

== Authentication and Invalidation

Identity, sessions, and token security follow
link:adr/0015-authorization-profiles-and-content-controls.adoc[ADR 0015] and
link:adr/0016-authentication-mechanisms-and-session-security.adoc[ADR 0016]: ES256-signed access
tokens with nested scopes, mandatory strict expiry, and digest-only rotating refresh tokens.
Ordinary API claims remain valid only until their short expiry; refresh revalidates live session,
account, and context authority. Playback additionally checks live authority on stream creation and
every playlist and segment request, as
link:adr/0018-live-playback-authority-and-outbound-workers.adoc[ADR 0018] specifies.
The browser client is same-origin by construction -- development
tooling proxies the API paths and production serves the client from the application origin --
so no CORS configuration exists and none should be added (recorded as a rejected alternative in
ADR 0016).

Tokens are signed with an EC P-256 key (`AUTH_TOKEN_SIGNING_KEY`, base64 PKCS#8; generate with
`openssl ecparam -name prime256v1 -genkey -noout | openssl pkcs8 -topk8 -nocrypt -outform DER |
base64`). Only the media server holds the private key; verifiers need public keys only, chosen
by the RFC 7638 thumbprint each token carries as `kid`. Rotation is two-phase so an HTTP cache
can never hide the incoming key: first add the new public key (base64 SPKI) to
`AUTH_TOKEN_VERIFICATION_KEYS` and restart -- it is prepublished in JWKS while the old key
keeps signing; then, after at least the JWKS `Cache-Control` window (5 minutes), configure the
new private key, replace the prepublished entry with the old public key, and restart --
outstanding tokens keep verifying until they expire. Skipping the prepublish wait risks
verifiers behind a shared cache rejecting newly signed tokens for up to the cache window,
because a refetch on unknown `kid` can be answered with the cached pre-rotation key set. A
blank signing key generates an ephemeral key pair at startup (tokens die on restart; sessions
recover through refresh).

ES256 exists for the planned transcode split: horizontally scaled transcode nodes running
FFmpeg are the most exposed tier and must never hold a token-minting secret. Signature
verification alone never authorizes a request, though -- revocation is authoritative state,
not signature checking. Playback revocation is one indexed live-authority query rather than token
counters or a persisted stream-session state machine. Workers never verify
playback tokens; the server remains the playback request gate and segment proxy.
The public verification keys are served at `/.well-known/jwks.json` (ADR 0016).

V049 removed the version claims, cache, PostgreSQL notification feed, counter columns, and
pooler-mode exception. Ordinary API tokens retain issued authority until strict expiry; refresh
checks the live authorization session and context, and every playback request checks live authority.

== Key Design Decisions

See link:adr/[Architecture Decision Records] for detailed rationale. Summary:

|===
| Decision | Rationale

| PostgreSQL over SQLite/H2
| Production-grade, supports custom ENUMs, UUID PKs, full-text search

| GraphQL over REST
| Flexible queries for rich tvOS client, Relay pagination for infinite scroll

| Keyset pagination over offset
| O(1) performance regardless of page depth

| ProcessBuilder + FFmpeg over JNI
| Full control over process lifecycle, proven pattern (Jellyfin/Plex/Emby)

| TDD and behavior testing
| Tests at service boundary are protocol-agnostic and survive refactoring

| TMDB metadata abstraction
| Provider interface allows future sources without changing the pipeline

| Spring Application Events
| Decouples side effects from core operations, eliminates circular dependencies (link:adr/0010-spring-application-events.adoc[ADR 0010])

| Stream decision architecture
| Per-stream decision records compose into TranscodeDecision; mode resolution considers all stream types (link:adr/0013-stream-decision-architecture.adoc[ADR 0013])
|===
