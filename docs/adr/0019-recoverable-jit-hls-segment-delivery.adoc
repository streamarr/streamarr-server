= ADR 0019: Recoverable Just-In-Time HLS Segment Delivery

== Status

Proposed

This ADR narrows the "RFC 8216 compliance" consequence of link:0008-server-generated-hls-playlists.adoc[ADR 0008] and refines link:0018-live-playback-authority-and-outbound-workers.adoc[ADR 0018] by un-deferring its request-driven recovery slice while leaving 0018's multi-control-plane failover and durable job recovery deferred.

== Context

Streamarr transcodes on-demand library files and serves them as HLS, generating the media playlist from the source's probed duration rather than from produced output.
The playlist advertises every segment up front and closes with `EXT-X-ENDLIST`, while FFmpeg produces those segments lazily behind it.
A request for an advertised-but-unproduced segment currently waits on a fixed wall-clock budget and then returns `404`, and a failed transcode is not recovered: the outbound-worker path abandons the variant with no re-dispatch, and the player cannot distinguish a permanently dead segment from one that is merely slow.

RFC 8216 §6.2.1 requires that any segment listed in a playlist be immediately downloadable, and forbids removing or altering a segment once it has been advertised.
Advertising the whole timeline up front is the price of instant start and arbitrary seek; it is an intentional deviation from §6.2.1's immediate-availability requirement, and it converts every advertised segment into a standing obligation to attempt recovery and deliver.
A listed segment cannot later be withdrawn, gapped, or substituted; exhausting a finite recovery budget is a failure to fulfill the obligation, not another valid representation of it.
Robust playback under that obligation is therefore not optional polish; having promised every segment, the server must be able to produce any of them.

== Decision

=== The advertised playlist is a delivery obligation

On-demand media playlists continue to advertise every segment computed from source duration, close with `EXT-X-ENDLIST`, and embed each segment's own seek offset in its URI so any segment is independently re-producible.
A listed segment is an obligation, not a hint: the server never un-advertises, gaps, or substitutes an advertised segment and normally fulfills the obligation by recovering a producer and delivering the segment bytes.
When every eligible execution target has been attempted once in the current recovery cycle and none can produce the segment, the controller returns `503 Service Unavailable`.
That terminal response reports failure to fulfill the advertised representation and gives the player a clear boundary at which to switch variants or stop; it is not treated as successful HLS delivery.
The advertise-ahead deviation from §6.2.1 is intentional and is recorded here rather than hidden.

=== Segment waits are bounded by producer liveness, not a wall clock

A segment is ready when it is atomically published under its final name.
The local FFmpeg path's temporary-file rename and the remote upload path's prepare-then-commit protocol provide that publication boundary; the existence of the following segment is not required.
For a segment store that cannot publish atomically, the following segment or successful producer completion may prove that the current segment is closed.
Successful completion therefore makes an existing tail segment ready, while failed completion does not make a possibly partial tail ready.

A request for an advertised-but-unproduced segment waits while a producer for that segment is alive and making progress.
Local process exit, remote-worker disconnect, and stalled progress are distinct liveness outcomes and are recorded separately, even though each can trigger recovery.
Stalled progress is determined by a bounded progress detector; it does not impose a fixed lifetime on the segment request while recovery remains possible.
The wait ends when the segment is ready, when the finite recovery cycle is exhausted, or on request cancellation — never merely because a fixed request wall-clock timeout elapsed.
A slow or resource-starved transcode must not surface to the client as a missing segment.

=== Producers are recovered on demand, driven by the request

When an advertised segment is requested and no live producer covers it, the server starts a fresh producer seeking to that segment; recovery is pull-based, triggered by the segment request, not a supervised push loop.
A recovered producer starts at the requested segment's advertised source offset and media-sequence number and emits a contiguous run rather than a single isolated segment.
That run is necessary but not sufficient for seamless recovery: the replacement must preserve the advertised media timestamp and decode timeline across the producer boundary, including MPEG-TS timestamps and continuity counters or the fMP4 base decode time, as applicable.
If a replacement cannot reproduce that boundary without an `EXT-X-DISCONTINUITY`, it cannot transparently fulfill the immutable playlist because that marker cannot be inserted after advertisement.
Seeking uses this same mechanism: a request beyond the produced window restarts a producer at the requested offset, and because segment numbering derives from the offset the playlist never changes.

=== Distributed producers are the recovery redundancy

When the transcode executor is distributed, a variant whose producer fails is re-dispatched, on the next request that needs it, to another available worker for the same source namespace; the pool of workers is the recovery redundancy.
Each recovery cycle snapshots the currently eligible execution targets and attempts each at most once.
Only after no eligible target can produce the variant does the controller return `503 Service Unavailable`, and the client's adaptive-bitrate switch to a healthy variant is the last-resort recovery.
A variant with no producible source therefore degrades at a bounded, documented boundary; it is not retried without limit, and the coincident-multi-failure case is accepted rather than engineered against.
Transient upload back-pressure is a self-classifying condition: the producer retries the affected segment with bounded back-off rather than failing the whole variant.

=== A playlist structure suite gates the advertised representation

A spec-cited in-repo suite runs inside the standard build and the continuous-integration gate and asserts the structure of generated playlists.
It does not claim to prove segment availability from playlist text, and the advertise-ahead deviation is recorded in this ADR rather than as a disabled, tautological test.
The recovery implementation tracked by issue 252 must add service-level behavioral tests with controllable producer and segment-store fakes for successful delivery, failover, cancellation, final-segment completion, and recovery exhaustion.
It must also add a small real-FFmpeg integration test that crosses a producer boundary and validates timestamp and container continuity; a fake cannot prove that property.
A deeper platform-specific stream validator remains an optional local check outside the portable gate.

* Explicitly deferred:
** cross-worker session affinity, durable job recovery, and multiple control-plane replicas — deferred by ADR 0018 and unchanged here;
** low-latency HLS (partial segments, blocking playlist reload, preload hints, delta playlist updates, rendition reports) — it targets live-edge latency, presumes a growing playlist, and is incompatible with the on-demand static-playlist contract; it would be revisited only if true live streaming is introduced, under its own ADR;
** a growing or event-type media playlist for on-demand content; and
** object-store delivery and any worker-to-client segment path.

== Consequences

The static-playlist obligation makes recovery mandatory rather than optional, and the decision states that obligation as a first-class contract.
Liveness-bounded waits remove the spurious-timeout failure mode under load and slow start for both local and distributed execution.
Pull-based recovery makes a failed producer invisible to the viewer in the common case: a brief stall while a fresh producer catches up; `503 Service Unavailable` appears only after the finite recovery cycle is exhausted.
The active playlist structure suite converts structural protocol assumptions into always-on checks; the delivery suite lands with the recovery implementation tracked by issue 252.
ADR 0008's blanket "RFC 8216 compliance" consequence is narrowed: server-generated playlists remain fully controlled, but on-demand delivery intentionally deviates from §6.2.1 immediate-availability as recorded here.
The costs are added producer-lifecycle and liveness logic in the streaming layer, per-request liveness evaluation on the segment path, continuity validation across producer boundaries, and additional build and continuous-integration time for the protocol suites.

== Rejected Alternatives

* Growing or event-type media playlist advertising only produced segments — satisfies §6.2.1 but forfeits instant whole-timeline seek, makes segment production stateful, and complicates recovery for no viewer-visible gain.
* Low-latency HLS blocking playlist reload as the segment-wait mechanism — presumes a growing live playlist, does not apply to a static on-demand playlist, and specifies no error path for a segment that cannot be produced.
* A fixed wall-clock segment-wait timeout returning `404` before liveness-driven recovery — converts slow production into spurious failures and papers over the missing recovery instead of providing it.
* Signaling a dead variant immediately through a distinct HTTP status or an advertised playlist gap — the standard forbids withdrawing a listed segment and the client cannot reconstruct or skip an advertised segment URL. A terminal `503 Service Unavailable` is reserved for finite recovery exhaustion; the playlist remains immutable.
* Per-segment isolated transcoding on demand — independent single-segment encodes glitch at audio and video boundaries; a contiguous run per producer is required.
