= ADR 0016: Authentication Mechanisms and Session Security

== Status

Accepted

=== Implementation status

* Version counters remain operative and load-bearing through PR-B1 of the revocation transition; they retire in PR-B2 (issue #213). Until then access and playback tokens continue to carry version claims.
* The persisted playback-enforcement lifecycle this ADR's revocation model relies on is specified in link:0017-playback-enforcement-lifecycle.adoc[ADR 0017] and lands with issue #211 (PR-A); PR-B1 integrates atomic revocation; PR-B2 completes the transition.
* Distributed transcoding topology A (issue #76) is the recommended direction, not ratified; a future ADR 0018 decides the transport.

== Context

link:0015-authorization-profiles-and-content-controls.adoc[ADR 0015] defines Streamarr's account, profile, household, authorization, and content-visibility model.
It assumes authenticated account/device sessions, profile-scoped request identity, token invalidation, and an authentication path that works across browsers, mobile clients, and limited-input TV apps.

This foundation should land before richer content-control enforcement — but not as a `userId`-only shortcut.
Its first increment must carry the account, household, profile, and session identity ADR 0015 needs, with authorization checks already behind the Streamarr authorization facade.

Authentication must stay independently revisable from that authorization.
Password-hashing work factors, MFA requirements, and device-pairing details will change as hardware, Spring Security, client capabilities, and security guidance move.

== Decision

Spring Security is Streamarr's authentication and security-context spine: it owns authentication plumbing, request authentication, method-security integration, context propagation, factor authorities, and future OAuth.
Streamarr domain authorization stays separate, as link:0015-authorization-profiles-and-content-controls.adoc[ADR 0015] describes.

Local password authentication comes first.
Store passwords with a custom `DelegatingPasswordEncoder` whose encoding id is Argon2id — not Spring Security's default, if that default still encodes with bcrypt.
Persist the full encoded string (algorithm id, parameters, salt, and hash) in the account's password column; never keep salts in a separate application-managed column.

Argon2id parameters are configuration, not architecture.
They must meet or exceed the current OWASP Argon2id minimum, be benchmarked on the target hardware, and keep login and setup latency practical for a self-hosted server — expensive enough to resist offline guessing, cheap enough for everyday login.
When `PasswordEncoder.upgradeEncoding` reports stronger parameters, rehash the password on the next successful login.

Plaintext passwords follow the framework-native `String`/`CharSequence` path across the Spring and Jackson request boundary and exist only for the synchronous setup, verification, or rehash operation that consumes them.
They are never persisted, cached, published in application events, logged, traced, included in exceptions, or retained by asynchronous work.
Every password-bearing request or command defines a safe string representation that omits the password, and generated builders are treated as secret-bearing objects that must never be logged.
This is a deliberate acceptance of best-effort JVM memory lifecycle, not a claim that immutable strings can be erased.

Copying a password into `char[]` or `byte[]` at only one application layer does not provide secure erasure when Jackson, Spring Security, or the password encoder has already created or will recreate an immutable `String`.
Streamarr adopts mutable password buffers only if the complete input-to-hasher path owns those buffers, avoids immutable copies, clears every owned intermediate in `finally`, and the threat model justifies the added custom security code.
Such a change replaces part of the Spring Security password-encoding boundary and therefore requires revisiting this ADR.

Password login is throttled: failed attempts are rate-limited per account within a sliding window, as NIST SP 800-63B requires for online guessing.
Per-source pressure is tracked and surfaced for alerting rather than hard-blocked: the normal self-hosted deployment sits behind a reverse proxy where every client shares one source address, so a blocking source budget would let five failed logins lock every account out server-wide; forwarded-header resolution is opt-in configuration for proxied deployments that want real client addresses in the signal.
Spring Security ships no login rate limiter, so this throttle is Streamarr code invoked inline before any password verification — reserving budget up front, rather than recording failure events afterwards, keeps a concurrent burst from overrunning it; the profile-PIN limiter shares this posture.

Account/device sessions use long-lived, revocable refresh tokens, scoped to the account and device rather than to a household or profile.
Refresh tokens rotate on every use: redeeming one consumes it and issues a successor, the server stores only a digest of the active token, and redemption of an already-consumed token is treated as theft and revokes the session's whole token family.
Requests use short-lived access tokens in one of three explicit scopes:

* *account-scoped* — carries account and session only; valid for login, session, listing the account's own household memberships, and selecting a household;
* *household* — issued when a household is selected or switched, or immediately at login if exactly one is selectable; valid for pre-profile and household-management surfaces that need no profile policy;
* *profile* — issued on profile selection and nested under the active household.

A profile token must name a profile that belongs to its household, and the server enforces that.
Scopes nest rather than exclude: a household token also reaches account-scoped surfaces, and a profile token also reaches the household- and account-scoped surfaces of its own household.
A client therefore holds one access token at a time and upgrades it by selecting a household or profile; nesting widens context, never privilege, because every surface still applies ADR 0015's role and policy checks.
The selected context is session state, not client state: the session row remembers the active household and profile, login auto-selection and the selection endpoints write it, and refresh re-derives scope from it only after a single account-profile lookup revalidates (account, stored household, stored profile) — selecting or switching a household always clears the stored profile first, so a stale selection can never mint a mismatched household/profile token, and failed revalidation downgrades to the highest valid scope.
In Spring Security terms, each scope becomes a granted authority and the nesting is a declared authority hierarchy — published as a `RoleHierarchy` bean that request-level and method-level authorization auto-detect — so request and method rules demand only the minimum scope.
Every access token authorizes requests only within the context chain it names, and carries enough account, session, and — where selected — household and profile identity for the server to validate the session and the ADR 0015 authorization model.
This makes access tokens self-contained and server-signed — validating one needs the public verification key set, not a database read — while refresh tokens stay opaque random secrets the server knows only by digest.

Token expiry is strict and mandatory.
RFC 7519 §4.1.4 makes `exp` optional; Streamarr makes it mandatory with zero leeway — a token is valid only while `now < exp` — because access-token expiry is the outer bound of the revocation model, and the default validator's clock skew and missing-`exp` tolerance would silently extend it.
A past-expiry failure keeps the machine-readable `EXPIRED_TOKEN` refresh-and-retry signal; a token without `exp` is malformed and maps to `INVALID_TOKEN`.
The configured access-token TTL validates against a fifteen-minute ceiling (ten-minute default): bounded API staleness is a security property configuration must not silently stretch.
A selection token is a new JWT whose expiry is capped by its source authorization — `exp = min(now + TTL, source exp)`, with the source expiry read from the validated token, never reparsed — so selection changes context, never authority, and repeated selection cannot extend access.
Only setup, login, refresh, and successful password reauthentication start a fresh TTL.
Playback tokens are not exempt from mandatory expiry: they carry a concrete `exp` at session-retention scale, as described below.

Signing keys are configuration: rotation starts issuing with a new key while old keys remain verification-only until every token they signed has expired, which short access-token TTLs keep brief.
Tokens are signed asymmetrically (ES256, P-256), each carrying its key's RFC 7638 thumbprint as `kid`: only the media server holds the private key, and verifiers need public keys only.
This is defense-in-depth and optionality rather than a present operational requirement: the recommended segment-serving topology for the transcode split (issue #76, topology A) proxies segments through the API server, so transcode workers never verify a playback token — but keeping the minting key private to the media server preserves worker-as-verifier and edge-verifier topologies without ever placing a signing secret beside FFmpeg and untrusted media.
Symmetric HMAC signing would be simpler and equivalent in a single process; it was set aside to keep those topologies open.
A public key is never sufficient authorization, though: signature-plus-expiry checking alone cannot see revocation — for playback that authority is the stream-session row and authorization session together (link:0017-playback-enforcement-lifecycle.adoc[ADR 0017]), so any future external verifier must check that joint authority or be handed short-lived signed URLs that encode the check.
The public verification keys are served at `/.well-known/jwks.json` (revisiting an earlier deferral: shipping ES256 without its discovery surface left the design incomplete, and the endpoint is pure public data).
It serves only public JWKs — current plus retired, each carrying `kid`, `use: sig`, and `alg: ES256` so verifier key selectors need no out-of-band knowledge; private material never appears in a response, and a test pins that.
The endpoint is permitAll and cookie-blind like the pre-auth endpoints (a stale `Path=/` access cookie must not 401 a key fetch); `Cache-Control: max-age` paces polling, and refetch-on-unknown-`kid` remains the rotation mechanism — which makes rotation two-phase: the incoming public key is prepublished through `AUTH_TOKEN_VERIFICATION_KEYS` at least one cache window before it starts signing, because a shared cache may answer an unknown-`kid` refetch with the pre-rotation key set (runbook in architecture.adoc).
Serving keys grants no authority: revocation is the condition for *using* them, and it binds every JWKS consumer — during the transition below that means the counter feed; after it, the joint playback authority of link:0017-playback-enforcement-lifecycle.adoc[ADR 0017].

Revocation runs on two channels, matched to what each surface can tolerate.
API surfaces accept bounded staleness: access tokens live only minutes, revocation prevents renewal, and access-token expiry bounds the residual API authorization window — an invalidated session's tokens keep ordinary API reach for at most one access-token lifetime, and the strict-expiry rule above makes that bound hard.
Role and context claims share that bound: they are honored as issued for the token's remaining life, which is current behavior and an accepted trade, not an oversight — content-policy enforcement against authoritative state is a deferred obligation tracked with the content-control work, not something claims-based authorization provides today.
Playback tolerates no staleness bound, because playback-token validity spans a media traversal plus a retention window; its per-request enforcement authority becomes the persisted stream-session row and the authorization session *together*, specified in ADR 0017 — a playback request is then served only while its stream row is `ACTIVE`, its authorization session is unrevoked, and its owning account is enabled (during the counter transition, the per-request version check carries this role, and account disablement does not yet terminate active playback).
Membership and profile-grant removal fail closed structurally: the stream row's composite foreign keys bind it to the grant chain, and parent deletion routes through the terminal-cleanup choreography rather than silently erasing enforcement state.
A password change immediately invalidates renewal and playback authority while the prior access token retains only its bounded ordinary API window.
Its target mechanism (PR-B1) is revoke-and-replace — the caller's old session and streams are revoked and a replacement session with fresh credentials is issued; during the counter transition the caller's session instead survives with a bumped version and a reissued refresh-token family, which invalidates the same credentials without a replacement session.

Until that transition completes (issue #213, PR-B2), restriction, membership, password, and session changes propagate through *version counters*, and this mechanism remains normative and load-bearing:
Postgres holds the authoritative session, membership, and profile-policy counters; each instance caches them in memory.
When an instance changes one, it bumps the Postgres counter and publishes the new version through `LISTEN`/`NOTIFY`; every other instance applies it to its cached counter monotonically (highest version wins), so duplicate or out-of-order delivery cannot regress state.
The `LISTEN` subscriber holds its own dedicated Postgres connection, separate from the transactional Hikari pool.
That connection must reach Postgres directly or through a session-mode pooler: `LISTEN` does not survive transaction-mode pooling, so PgBouncer in transaction mode silently breaks notifications.
On instance start and whenever the notification connection is unavailable, caching is suspended and every counter lookup reads through to Postgres; successful reads are not installed until `LISTEN` succeeds again, so a missed notification cannot strand a stale refill.
Once the feed is listening, the empty cache fills from Postgres on first touch and each request checks its token's embedded versions against the in-memory value, so the healthy hot path needs no database round-trip.
When an authoritative counter cannot be read, validation fails closed rather than trusting a token's embedded versions.
Retiring the counters removes this machinery, the version claims, and the session-mode-pooler constraint; there is no point in the transition where playback lacks both counter validation and persisted enforcement.

How a client presents tokens depends on what it can protect.
Native apps — iOS, Android, TV — send access tokens as `Authorization: Bearer` headers and keep refresh tokens in platform-secure storage.
Browsers never touch tokens from script: both tokens travel as `httpOnly`, `Secure`, `SameSite` cookies, with the refresh cookie path-scoped to the refresh endpoint.
Cookies are ambient credentials, so cookie-authenticated requests keep CSRF protection — a manually wired `CsrfFilter` pairing the BREACH-protected `XorCsrfTokenRequestAttributeHandler` with `CookieCsrfTokenRepository`, applied by a matcher that exempts only bearer-header requests, which carry no ambient credential; the `csrf.spa()` DSL was deliberately not used because it would exempt cookie-authenticated requests as well.
`httpOnly` keeps scripts from reading the session cookie, and cross-site tracing cannot echo it back while HTTP `TRACE` stays disabled — embedded Tomcat's default, which must not be re-enabled.
That narrows XSS rather than eliminating it: an injected script can still ride the session, so Spring Security's default security headers, a Content-Security-Policy, and output encoding remain required hygiene.
Streaming is the third carrier: playlist-driven players — AVPlayer above all — cannot attach headers to the segment requests a playlist spawns, so issued manifest, playlist, and segment URLs embed a playback-scoped, stream-session-bound profile token the server validates exactly like a header token.
Same tokens, different carrier, not a weaker channel: playback-URL tokens authorize nothing but playback, and the server redacts them from its own request logs and traces.
Logs the server does not own — a reverse proxy in front of a self-hosted install — will record these URLs, which is exactly why the token's playback-only scope and binding to a single stream session bound what a captured URL is worth.
Playback-token validity is the probed media duration plus the configured session-retention window — one full traversal plus a bounded cumulative budget for pauses and slow playback, rather than a fixed minutes-scale TTL; this is the concrete `exp` mandatory expiry demands.
VOD players fetch playlists once and then request segments for hours, so the token is not renewed from its own authority; the absolute expiry remains the captured-URL exposure backstop, while revocation stays instant through per-request enforcement — the version check during the transition, the ADR 0017 joint authority after it — and idle session reaping may end playback sooner.

Renewal is a server contract before it is client code.
The server never slides an access token or re-issues one on its own authority; renewal always passes through the refresh endpoint — except the password-change flow, which requires password reauthentication and therefore earns its fresh credentials.
The whole refresh use case is one transaction: redemption joins it, the session lock taken during redemption is held through issuance — a concurrent logout waits, and a logout that committed first refuses redemption — a disabled account is refused, and any failure after rotation rolls the rotation back instead of stranding the client's token family.
Redeeming a refresh token at the dedicated refresh endpoint rotates it, sets the successor cookies where cookies are the carrier, and — like login — returns the access token's expiry in the response body, since script cannot read an `httpOnly` cookie to schedule ahead.
An expired access token fails with a machine-readable expired-token code distinct from other authentication failures, so a client knows refresh-and-retry is worthwhile.
Rotation's theft rule tolerates one honest race: every tab of a browser shares one cookie jar, so concurrent refreshes can innocently redeem the same token.
Each rotation's successor is deterministic but secret: the server derives it with HMAC-SHA-256 keyed by the presented 256-bit predecessor over a domain label and that predecessor's internal refresh-token row UUID, then stores only the successor's digest.
Within a short grace window — judged from the persisted rotation timestamp, so it survives restarts and works across instances — the server derives the same successor again and returns it only while its digest still names the session's active, unexpired token.
This makes a lost successful rotation response recoverable without storing plaintext, while concurrent responses can only write the same active successor into the browser's cookie jar.
If that successor has already rotated, the older predecessor is superseded: its grace replay may receive a fresh access token but no refresh token, so it cannot overwrite the current cookie with a consumed credential.
Exactly one genuine rotation happens per predecessor, and reuse outside the window stays theft.
On the web, a service worker owns renewal: one worker serves every tab of the origin, making it the natural single-flight point — it intercepts the app's API fetches, refreshes once when expiry nears or an expired-token response arrives, queues intercepted requests, and replays them once the successor cookies land.
The worker coordinates timing but never handles the tokens themselves; the cookies stay `httpOnly` and ride its fetches automatically, and the page supplies the script-readable CSRF token the refresh `POST` needs, since workers cannot read `document.cookie`.
Browsers kill idle workers and a first visit precedes registration, so correctness never rests on the worker: the grace window and expired-token retry keep a workerless tab correct, and the worker's job is making renewal invisible.

Limited-input clients — Apple TV, Android TV, Nvidia Shield, smart-TV apps — pair rather than type credentials.
The TV requests a short-lived, single-use device code and shows it to the viewer; the code is only a correlation handle for a pending authorization, not a credential.
An already-authenticated browser or mobile session approves the device; the server accepts the approval only from a session whose login already satisfied the account's effective MFA policy, so pairing rides the approver's existing factors without a fresh challenge.
Spring Security makes that check declarative: satisfied factors surface as `FactorGrantedAuthority` authorities, and the approval endpoint requires the ones the account's policy demands.
This flow authenticates the account/device session and may use Spring Security one-time-token login or an equivalent.

It is not profile switching.
A profile PIN is a Streamarr domain gate for selecting a profile on an already-authenticated session — not an account credential, one-time token, or MFA factor.
PIN verification is server-side and rate-limited with lockout or backoff, since PINs are low-entropy convenience gates.

Capable clients — browser, iOS, Android — authenticate directly, using local password auth, future OAuth, and MFA per the account's effective policy.
Every client receives tokens only after its required factors are met; TV clients are not weaker, they delegate credential entry and MFA to a capable authenticated device.

Authentication policy is admin-managed.
Admins set minimum MFA requirements for all accounts, admin accounts, or specific accounts, and later for sensitive actions.
A user may choose stronger MFA but never fall below policy; the effective requirement is the strongest of the server, account, and action policies.
The first modes are `OFF`, `ADMIN_ONLY`, and `ALL_USERS`, with action-specific MFA deferred but allowed for in the model.
Spring Security 7 carries this model natively: `AuthorizationManagerFactories.multiFactor()` requires factor combinations globally or per endpoint, `RequiredAuthoritiesRepository` resolves per-account requirements, and per-factor `validDuration` leaves room for the deferred action-specific step-up.

WebAuthn/FIDO2 is the first MFA factor — a second factor, not passwordless passkey login yet; when it is required, authentication needs both password and WebAuthn.
TOTP is the planned fallback factor.
Recovery codes are required once MFA exists so users and admins keep a safe recovery path.
Email or magic-link one-time tokens may support recovery or device-pairing UX but are never the primary factor.

Future OAuth providers map onto local `UserAccount` records and inherit the same account, household, profile, session, MFA, and token model.

Explicitly deferred from v1:

* Passwordless passkey login — WebAuthn ships as a second factor first
* TOTP as the fallback second factor
* Action-specific, step-up MFA — the policy model allows for it, nothing builds it yet; device-pairing approval is the natural first target
* OAuth provider login

== Consequences

* Authentication can evolve without reopening ADR 0015's authorization and content-visibility decisions.
* Password-hashing choices stay behind Spring Security's `PasswordEncoder`, so algorithm and work-factor upgrades need no schema change.
* Rainbow-table resistance comes from the encoder's per-password salts, not custom code.
* Argon2id work factors live in configuration or a security runbook, not this ADR.
* Plaintext passwords may remain in JVM memory until garbage collection; minimizing their lifetime and preventing logging, tracing, diagnostic capture, and asynchronous retention are the controls for the accepted framework-native path.
* A redacted `toString()` prevents accidental disclosure but does not erase a password from memory.
* MFA and device pairing are account/session concerns; profile selection stays a viewing-context concern.
* Refresh-token revocation, rotation with reuse detection, digest-at-rest storage, and playback-authority revocation are required server features, not optional client behavior.
* One token model has three carriers — bearer headers for native apps, `httpOnly` cookies plus CSRF protection for browsers, session-bound playback-URL tokens for streaming — and no carrier weakens validation.
* Access-token `exp` is inviolable and the TTL ceiling is a security parameter: bounded API staleness is only sound while both hold.
* API-surface revocation is bounded by the access-token TTL; instant revocation is a playback property, carried by the joint authority of link:0017-playback-enforcement-lifecycle.adoc[ADR 0017] once the counter transition completes.
* Refresh races are resolved on the server — the rotation grace window — and merely smoothed by the web client's service worker, so a browser without a live worker degrades to slower renewal, never to spurious logout.
* Login and PIN throttling are Streamarr code that reserves budget inline before password verification, not framework features.
* Cross-instance invalidation rides Postgres version counters and `LISTEN`/`NOTIFY` during the transition, adding no Redis or other new infrastructure; after PR-B2 revocation state rides ordinary PostgreSQL rows and the pooler-mode constraint disappears.
* While the counter feed is live, the invalidation listener must never borrow and hold a connection from the transactional Hikari pool, even while reconnecting or backing off.
* OAuth providers are authentication inputs to local accounts, not replacements for the account/profile/household model.

== Rejected Alternatives

* Spring Security's default `DelegatingPasswordEncoder` id — it still encodes with bcrypt, OWASP puts Argon2id first, and the delegating mechanism makes the stronger default free.
* An application-managed salt column — the encoded password string already carries algorithm, parameters, and salt; a separate column invites custom crypto handling.
* Converting only service-layer passwords to `char[]` or `byte[]` — surrounding framework layers retain or recreate immutable strings, adding copies without delivering end-to-end erasure.
* Redis or any shared cache for cross-instance invalidation — Postgres version counters with `LISTEN`/`NOTIFY` cover v1 without new infrastructure.
* CORS configuration for the browser client — the client is same-origin by construction (development tooling proxies API paths; production serves the client from the application origin), so no CORS policy exists and adding one would only widen the attack surface.
* Sliding or self-renewing access tokens — re-issuing near expiry on the access token's own authority makes the short-lived credential immortal for whoever holds it, yields no rotation theft signal, lets active clients outrun the TTL backstop for missed invalidations, and skips the refresh endpoint's authoritative database checkpoint.
* Short access-token TTLs as the sole revocation mechanism *for playback* — bounded staleness is accepted for API surfaces, but playback-URL tokens have no refresh channel: playlist-driven players fetch segment URLs once and replay them for hours, so playback validity must ride the stream-session lifetime and only a per-request authority check can revoke it mid-stream; TTL stays a backstop, never the guard.
* Version counters as the permanent revocation authority — playback is their only load-bearing case once the stream-session row exists, one authority beats two, and retiring the feed removes the version claims, the dedicated-listener plumbing, and the session-mode-pooler constraint; they remain normative only until PR-B2 lands.
* Script-readable browser token storage (`localStorage` or plain cookies) — `httpOnly` cookies plus CSRF protection confine XSS to riding a session rather than stealing it.
* `Authorization` headers on streaming segment requests — playlist-driven players cannot attach them, so playback URLs carry the token instead.
* Profile PINs as an authentication or MFA factor — low-entropy convenience gates guard viewing context, never accounts.
* Weaker factor requirements for TV clients — pairing delegates credential entry and MFA to a capable device instead of lowering the bar.
